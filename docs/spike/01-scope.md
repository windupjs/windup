# Validation Spike — Scope

## Objective

Validate the project's central hypothesis: **a browser test flow can be executed with an LLM only in planning (and in re-planning on failure), with deterministic execution and cheap verification — resulting in repeated executions with zero LLM calls.**

The spike exists to answer three questions, in this order:

1. **Feasibility** — Can Gemini generate a valid, executable JSON action plan from a task description + page context (via Stagehand v3)?
2. **Determinism** — Can the generated plan be re-executed (replay) without an LLM, with post-condition verification passing stably?
3. **Economics** — What is the measurable difference in latency and cost between the 1st execution (with LLM) and subsequent ones (cache replay)?

If all three answers are positive, the project is validated to proceed to the MVP.

## What is IN scope

- Minimal CLI (`spike run <scenario>`) — no graphical interface.
- Stagehand v3 running local headless Chrome/Chromium.
- JSON plan generation via **Gemini** (1 call per cache miss).
- Deterministic plan executor (loop over actions, no reasoning).
- Post-condition verifier (`expect_selector`, `expect_url`).
- Trajectory cache in a **local JSON file** (minimal persistence; Redis/SQLite is deferred to the MVP — see [04-schemas.md](04-schemas.md)).
- Per-execution metrics collection: total duration, per-action duration, input/output tokens, estimated cost, cache hit/miss.
- 2 scenarios on [saucedemo.com](https://www.saucedemo.com) — see [02-scenarios.md](02-scenarios.md).
- Reproducible environment via Docker — see [05-environment.md](05-environment.md).

## What is OUT of scope (explicitly)

| Out | Why |
|---|---|
| Graphical interface / dashboard | Validation is via CLI + metrics files |
| Selector self-healing | It is an optimization; the spike only needs to prove replay on the happy path + re-planning on failure |
| Partial re-planning with DOM diff | In the spike, a verification failure invalidates the plan and re-plans the entire flow (1 extra call). Partial diff is an MVP refinement |
| Robust page signature | Simplified cache key: `(scenario id, initial URL)` |
| Rich test assertions / reporting | Only action post-conditions and a metrics JSON |
| Multiple LLM providers | Gemini only. Provider abstraction is deferred to the MVP |
| Parallelism, sophisticated retries, screenshots | Screenshot fallback is not needed: saucedemo is 100% DOM |

## Spike success criteria (summary)

The spike is considered **validated** if, for the 2 scenarios:

- The plan generated by Gemini executes to completion with all post-conditions passing in ≥ 4 of 5 generation attempts.
- Replay of the cached plan passes in **10/10 consecutive executions** with **zero LLM calls**.
- Replay is **≥ 5x faster** than execution with plan generation and has **LLM cost = 0**.
- When a selector is deliberately broken, the system detects the failure via the post-condition and re-plans successfully.

Detailed metrics and procedure in [06-validation-criteria.md](06-validation-criteria.md).

## Learning non-goals

The spike does **not** attempt to measure Gemini's quality vs. other models, nor Stagehand's performance vs. plain Playwright. It only validates the JSON-plan + verification + cache architecture.
